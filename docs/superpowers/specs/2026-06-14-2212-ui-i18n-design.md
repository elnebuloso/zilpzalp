# Design: UI-Mehrsprachigkeit (i18n)

Status: abgestimmt · Datum: 2026-06-14

Backlog-Bezug: „Mehrsprachigkeit der UI" (docs/backlog.md → Ideen). Bei Aufnahme
in die Umsetzung gemäß Pflege-Regel als Zeile in die Umsetzungs-Tabelle übertragen
(Art: `Feature`).

## Ziel

Die Oberflächentexte werden aus den Templates und dem Python-Code gelöst und über
einen schlanken i18n-Mechanismus bereitgestellt. Zum Start werden **Deutsch und
Englisch** vollständig übersetzt. Die Sprache wählt der Nutzer über einen Umschalter
im Header; die Wahl wird per Cookie gemerkt, beim Erstbesuch entscheidet der
Browser (`Accept-Language`).

## Entscheidungen (abgestimmt)

1. **Sprachumfang:** DE + EN, beide vollständig.
2. **Sprachwahl & Persistenz:** Umschalter im Header + Cookie. Erstbesuch:
   `Accept-Language` → Fallback Deutsch.
3. **Mechanismus:** Leichte JSON-Kataloge + `t()`-Helfer (kein Babel/gettext,
   kein Build-Schritt). Begründung: ~50 Strings, „Simplicity first". Babel kann
   später eingezogen werden, falls die Sprachzahl wächst.
4. **Datumsanzeige:** Rechnungsdatum wird als normalisiertes ISO `YYYY-MM-DD`
   angezeigt — sprach-unabhängig. Der `german_date`-Filter entfällt.
5. **Renaming-Fix (mit drin):** Das Datum für die Dateiumbenennung kommt konsistent
   aus `config.date_format` — auch für vorgeschlagene Kandidatendaten (heute umgehen
   diese das Format).

## Architektur & Module

Neues Modul `backend/src/zilpzalp/web/i18n.py` plus Katalogdateien
`backend/src/zilpzalp/web/locales/de.json` und `en.json` (flacher Key→Text-Baum).

`i18n.py`:

- `SUPPORTED = ("de", "en")`, `DEFAULT = "de"`
- Kataloge werden beim Import aus `locales/*.json` geladen.
- `translate(key, lang, **vars) -> str` — Lookup in `lang`, Fallback auf `DEFAULT`,
  dann auf den Key selbst; `str.format(**vars)` für Platzhalter (z. B. `{filename}`).
- `resolve_language(request) -> str` — liest Cookie `lang`; fehlt es, wird der
  `Accept-Language`-Header gegen `SUPPORTED` gematcht; sonst `DEFAULT`.

**Anbindung in `routes.py`:** `_base_context()` ist der zentrale Punkt für alle
Seiten- und Partial-Routes. Dort wird `lang` aufgelöst und ein gebundenes
`t = lambda key, **v: translate(key, lang, **v)` sowie `lang` in den Context gelegt.
Routen mit eigenem Context (`_summary_response`) und Python-seitige Flash-Texte
(`_execute`, Fehlerpfade) rufen `resolve_language(request)` direkt auf.

## Sprachwahl (Umschalter + Cookie)

- Neue Route `GET /lang/{code}?next=<pfad>`: validiert `code` gegen `SUPPORTED`,
  setzt Cookie `lang` (langlebig, `SameSite=Lax`), 303-Redirect auf `next`
  (Default `/`).
- Umschalter im Header (`base.html`) neben dem Theme-Toggle: Links `DE` / `EN`,
  aktive Sprache markiert, `next` = aktueller Pfad (`request.url.path`). Funktioniert
  ohne JS, voller Reload.
- `<html lang="{{ lang }}">` statt fest `"de"`.

## Text-Externalisierung

- **Templates:** alle deutschen Strings → `{{ t("...") }}`. Keys nach Bereich
  gruppiert: `nav.*`, `overview.*`, `queue.*`, `review.*`, `config.*`, `status.*`,
  `original.*`, `summary.*`, `toast.*`, `action.*`.
- **routes.py-Konstanten:**
  - `STATUS_META` wird gesplittet: CSS-Klassen bleiben als `STATUS_BADGE` in Python,
    die Labels wandern in die Kataloge (`status.pending` …). Templates:
    `t("status." ~ entry.status)`.
  - `ORIGINAL_LABEL` / `SUMMARY_LABEL` → Kataloge (`original.move`,
    `summary.always` …).
  - Flash-Texte → Keys mit Platzhaltern, z. B.
    `toast.filed = "„{filename}" wurde abgelegt." / "“{filename}” has been filed."`,
    `toast.file_error`, formatiert via `translate(..., lang, filename=…)`.

## Datum (Anzeige + Renaming-Fix)

- **Anzeige:** `german_date`-Filter wird entfernt (Funktion + Registrierung). In
  `review.html`, `_queue_list.html`, `_overview.html` wird der normalisierte
  ISO-Wert direkt ausgegeben (`{{ c.normalized }}` bzw. `{{ preselected_date(...) }}`).
- **Renaming-Fix:** `_normalize_date` wird vereinheitlicht — sowohl Kandidaten- als
  auch manuelles Datum liegen als ISO `YYYY-MM-DD` vor und werden konsistent via
  `config.date_format` formatiert:

  ```python
  def _normalize_date(date_value, config):
      try:
          parsed = datetime.datetime.strptime(date_value, "%Y-%m-%d").date()
      except ValueError:
          return ""
      return parsed.strftime(config.date_format)
  ```

  Damit kommt das Renaming-Datumsformat immer rein aus der Config. Bei der Umsetzung
  wird geprüft, ob `date_kind` außerhalb der Formatierung noch eine Rolle spielt.

## Tests (an `tests/test_routes.py`-Muster angelehnt)

- `i18n.translate`: Lookup, Fallback DEFAULT, Fallback Key, Platzhalter-Interpolation.
- `i18n.resolve_language`: Cookie-Vorrang, `Accept-Language`-Parsing, Fallback DE.
- **Katalog-Vollständigkeit:** `de.json` und `en.json` haben exakt dieselben Keys
  (verhindert Drift).
- Route-Test: Cookie `lang=en` → englischer Text im gerenderten HTML; `/lang/en`
  setzt Cookie + redirect.
- `_normalize_date`: Kandidatendatum wird jetzt via `config.date_format` formatiert.

## Scope-Grenzen (bewusst draußen)

- Inhalte aus `config.yaml` (Pattern-/Target-Labels) bleiben Nutzerdaten, werden
  nicht übersetzt.
- Kein Babel, keine `.po`-Tooling.
- Keine Pluralregeln (nicht benötigt).
